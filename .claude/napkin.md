## User Preferences
- Keep fixes pragmatic and directly deployable for EasyPanel Docker builds.
- Default to TanStack Query for all client-side async fetch/mutation flows (avoid ad-hoc `useEffect` fetch state unless no query abstraction fits).
- Tulip integration should follow `1 API key = 1 renter`; renter UID must be auto-selected at connect time and not user-configurable in settings UI.
- Tulip integration should use a single activation state derived from connection status (avoid separate “enabled” and “connected” steps in UI flows).
- Tulip integration now uses a platform-level API key (`TULIP_API_KEY`) and store-level UUID input; do not reintroduce per-store API key storage/encryption flows.
- On Tulip disconnect/removal, preserve renter UID context for legacy contracts (at least as archived metadata/snapshot) so historical contracts remain traceable/manageable.
- Tulip product fetches must be filtered by the current store renter UUID (`product.uuid`, fallback `product.uid`) so each store only sees its own Tulip products.
- Tulip contracts must be created only when a reservation is accepted/confirmed, then kept in sync on reservation edits (date/product changes) with update/recreate flow.
- Tulip contract creation trigger ownership: dashboard/admin validation flows only (including dashboard edge cases), never storefront success-page fallback or Stripe webhooks.
- For Tulip product migration mapping, if a selected Tulip product is not Louez-origin, clone it into Tulip with a Louez origin marker and map to the clone instead of mapping directly to the external product.
- Tulip category/subtype selectors in product assurance dialogs should be driven by `renter.options.products` from `/renters/{renterId}` (dynamic), with static lists only as fallback.
- Apply the same dynamic Tulip category/subtype catalog to both product edit assurance dialog and integrations mapping sheet UI, not only one screen.
- In Tulip product mapping lists, append `(Louez)` to products created from Louez so users can identify origin at a glance.
- In Tulip integration settings UI, hide/remove global assurance configuration controls when these options are intended to be set per reservation flow.
- For Tulip settings, keep the configuration section but remove contract-type selector from dashboard UI; storefront insurance flow must validate selected contract type against renter API flags (`LCD/LMD/LLD`).
- In Tulip settings UI, remove the “prix inclus / include in final price” toggle because pricing inclusion is handled directly by Tulip contracts.
- On dashboard reservation edit, when period/items change and Tulip insurance is enabled, recalculate Tulip insurance amount and update the legacy insurance line item (`Garantie casse/vol`) to match the new quote.
- For dashboard reservation edit with Tulip insurance opt-in, if Tulip contract sync fails, rollback reservation + items to previous state and return an error (do not persist partial edits with insurance enabled).
- For Tulip contract updates, apply progressive PATCH payload fallbacks (full -> no `start_date` -> no identity -> minimal `end_date`/products -> `end_date` only) to handle Tulip immutability constraints (e.g. started >1h) without recreating contracts.
- Keep `AGENTS.md` clean and explicit; favor one `useTranslations` instance per file and reuse existing project logic before adding abstractions.
- Keep page files clean by moving pure/non-UI helper functions into colocated `utils` modules when practical.
- Prefer smaller files by default: keep page/form files orchestration-focused and extract hooks/sections once a file becomes large.
- For helper placement conventions, document and follow: inline if tiny/JSX-local, colocated `utils` first, then `lib/utils/util.<name>` for shared cross-feature helpers.
- For auth form async flows, prefer `useMutation` over ad-hoc async state handlers, and prefer local `useTranslations` usage in components over passing translators as props.
- For feature-local client hooks that render UI-related copy, prefer calling `useTranslations()` inside the hook rather than threading `t` callbacks through parent components.

## Patterns That Work
- For storefront date pickers, decide same-day eligibility from store rental rules (`minRentalMinutes < 1440`, with hourly pricing as fallback), not from shared page `pricingMode` alone; home/catalog/rental pages can run with `pricingMode='day'` while still allowing valid same-day rentals.
- For storefront same-day return time popovers, filter available end slots from the real minimum end datetime (`pickup + minRentalMinutes`) instead of only hiding slots `<= startTime`; otherwise invalid same-day return times still appear selectable.
- Production store cloning to dev is handled by `packages/db/src/scripts/store-clone.ts` with the package script `pnpm --filter=@louez/db run store:clone`; source DB comes from `--source-db-url`/`SOURCE_DATABASE_URL`, target DB from `--target-db-url`/`TARGET_DATABASE_URL` or fallback `DATABASE_URL`.
- `store-clone` should clone the logical `subscriptions` row so plan-based limits/features survive dev copies, while clearing `stripe_subscription_id` and `stripe_customer_id` to avoid linking dev stores to real Stripe objects.
- Store clones must also include `product_seasonal_pricing` and `product_seasonal_pricing_tiers`; otherwise dev storefront pricing can diverge sharply from prod even when base products and tiers were cloned correctly.
- Seasonal rate-based pricing intentionally forces progressive interpolation within each seasonal segment (`packages/utils/src/pricing/seasonal.ts`) even when the product itself is configured as strict tiers; compare seasonal data before assuming a strict/progressive bug.
- Storefront checkout and dashboard reservation delivery flows all share `apps/web/components/ui/address-input.tsx`; fixing typed-address resolution there removes the “must click the map” failure mode across all of them.
- In pnpm monorepos, prepare a dedicated runtime dependency folder in Docker build stages when scripts run outside Next standalone tracing.
- Cleaner OSS deployment model: split into `migrator` and `app` Docker targets; keep app image standalone and run migrations as a separate one-off job.
- Coerce SMTP booleans/ports at transport creation to avoid deployment env string edge cases (`"false"` becoming truthy in Node).
- For pre-migration pricing checks, use a legacy-only SQL preflight script (no reads of future columns) so checks can run safely before additive DB migration.
- For complex TanStack form UIs spanning multiple fields, extract to a dedicated `components/form/*` component and keep `form.Field` render props inside it (instead of bloating page-level JSX).
- For reusable choice UIs, build a generic `form-radio-card-group` and keep domain wrappers (like pricing mode) as thin option mappers.
- When the team wants maximal reuse, consume `form-radio-card-group` directly from pages and remove single-use wrappers to reduce component sprawl.
- For browser-derived onboarding defaults, initialize deterministic fallback values and then set `country` + `currency` in a `useEffect` to avoid server/client locale drift.
- For server-side field errors in TanStack Form, write to `field.meta.errorMap` (typically `onSubmit`) and clear that key when the user edits the field.
- For clean page files, extract pure helpers and use `apps/web/lib/utils/util.<name>.ts` once logic is promoted beyond a single feature.
- For large repo-wide path rewrites, run a verification grep immediately after scripted replacements to catch partial replacements early.
- For multi-step onboarding, expose a small server-backed draft query and hydrate each step once to prevent data loss when users go back/forward.
- For idempotent onboarding create/update actions, resolve the editable record first, then run uniqueness checks against "other records only" to avoid false slug conflicts.
- For oRPC in this Next.js setup, prefer an absolute RPC endpoint URL over relative paths to avoid runtime `URL` constructor failures on client mutations/queries.
- For dashboard image-crop UX, `react-easy-crop` works without importing a global CSS file in this setup (automatic style injection is sufficient), which avoids Next.js global CSS import constraints in feature components.
- For portrait-friendly fixed-aspect cropping, keep image `zoom=1` with `objectFit="contain"` and resize `cropSize` from a slider based on rendered-image bounds, not full container bounds.
- For Stripe subscription checkout, use `payment_method_collection: 'if_required'` when starting with trial days so zero-due sessions don’t force card entry.
- For onboarding steps that need role-gated UI in a client component, keep `page.tsx` as a server wrapper and pass role/store props to a dedicated client renderer component.
- For long `Command`/autocomplete dropdowns, set `keepHighlight={false}` to avoid reopening scrolled to the previously highlighted item.
- When switching active stores from dashboard pages, navigate to the dashboard section root (`/dashboard/<section>`) instead of refreshing full dynamic routes to avoid cross-store 404s on entity IDs.
- Store switching can leave stale dashboard data visible when query keys don’t include store scope; a full navigation (`window.location.assign`) after store cookie switch reliably resets TanStack cache.
- When `git status` shows `UU` but files contain no conflict markers, inspect with `git diff --cc` and resolve by staging (`git add`) the already-merged working tree file.
- For safety-critical integration triggers (like Tulip contract creation), require an explicit typed source parameter and validate it at runtime to prevent accidental calls from non-owned flows.
- Pricing currently flows through multiple calculation entrypoints (`calculateCartItemPrice`, dashboard new reservation hook, server reservation actions, storefront product/date widgets); compare them before changing pricing behavior because strict/progressive handling is not fully centralized.
- Storefront cart lines persist a pricing snapshot in localStorage, and multiple add-to-cart entrypoints (product page, product card, product modal, accessories modal) must all carry the same pricing fields (`basePeriodMinutes`, `enforceStrictTiers`, tiers) or cart pricing drifts from current product config.

## Domain Notes
- `docker/migrate.mjs` runs at container startup and imports `drizzle-orm/mysql2` plus `mysql2/promise`.
- `.npmrc` currently has `public-hoist-pattern[]=zod`, so many dependencies are only linked under workspace `node_modules`.
- EasyPanel UI may not expose Docker build target; provide dedicated Dockerfile paths for app and migrator services.
- `@louez/ui` `radio-group.tsx` exports `Radio`, and `RadioGroupItem` is just an alias of `Radio`; the card-label pattern can use `<Label><Radio .../></Label>` with `has-data-checked:*` classes.
- Tulip product creation (`POST /products`) expects a client-generated `uid` plus `product_type`, `title`, `data`, and `value_excl`; using this exact shape avoids avoidable 4xx errors.
- For Base UI `Select`, leaving value `undefined` (instead of sentinel values like `__none__`) renders placeholder text correctly and avoids leaking internal sentinel strings into the trigger.
- Storefront customer-login code emails are sent from `apps/web/app/(storefront)/[slug]/account/actions.ts` and currently swallow send failures (returning success to UI); failures only surface in server logs/`email_logs` when available.
- Runtime SMTP sender expects `SMTP_FROM` (`packages/email/src/env.ts`), while README examples still document `EMAIL_FROM`.
- Storefront OTP delivery can appear "broken" when SMTP provider delays inbound delivery; verify `email_logs` status + provider queue/deliverability before assuming app-side failure.
